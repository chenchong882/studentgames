# Student Games 介面易用性統一計畫

> 建立時間：2026-07-14（Asia/Taipei）
>
> 給實作者：開工前先讀根目錄 `AGENTS.md`、`CLAUDE.md`，並遵守 `studentgames-dev` skill。
>
> 本文件只規劃 UI／操作流程，不在本次重做題庫注入鏈、難度參數或遊戲玩法。

## 0. 本次目標

處理學生從教材頁進入 Student Games 後最容易卡住的 3 個問題：

1. 主機的「本次段考單字」與「系統內建單字」看不出差異，也不容易確認目前選了哪一個。
2. 各遊戲開始前的難度與「全圖檔模式」排列、名稱、操作方式不一致，學生需要重新理解。
3. 返回中央遊戲選單的按鈕位置不一致；狙擊手甚至同時有頂部與畫面底部兩個返回入口。

## 1. 規格優先順序

本計畫是現行介面的新 UI 規格，會取代 `PLAN-三模式改版.md` 中以下舊規則：

- 舊規則：「簡單／一般／困難」三顆按鈕永遠顯示，不可用的模式灰化。
- 新規則：**只顯示目前題庫真正可選的模式**，不讓學生先看到一個不能按的選項。
- 舊規則：「全圖檔模式」checkbox 放在模式按鈕前後，各遊戲自行安排。
- 新規則：統一改名為「題目內容」，使用明確的二選一控制。

`PLAN-三模式改版.md` 內已定案的題型比例、玩法難度、圖片題方向、`sgAllPic` 儲存鍵與 iOS 發音規範仍然有效，不要重新設計。

## 2. 主機題庫來源：配色與狀態直覺化

修改檔案：`index.html`

### 2.1 視覺語意

兩個來源不可再共用同一個黃色 active 樣式：

| 題庫來源 | 圖示 | 主色 | 輔助文字 |
|---|---|---|---|
| 本次段考單字 | 📝 | 橘色／暖色 | 老師這次指定的單字 |
| 系統內建單字 | 🎨 | 藍色／青色 | 可選主題，包含圖片題 |

顏色只做輔助，不能只靠顏色表達狀態。選中的來源必須同時具備：

- 明顯實心底色；
- `✓ 已選擇` 或勾選圖示；
- `aria-pressed="true"`；
- 鍵盤 focus 樣式。

### 2.2 排列

- 桌機：兩張等寬來源卡並排。
- 手機直式：兩張卡上下排列，整張卡都可點。
- 段考單字不存在時：保留卡片但顯示「需從學生教材頁進入」，不可點；不要只用低透明度讓人猜。
- 選內建題庫後，主題選擇區緊接在來源卡下方；選段考題庫時隱藏。
- 單字預覽按鈕放在目前來源摘要旁，不要混在兩個來源選項中間。

### 2.3 不可改動

- 不改 `lessonData` hash、`sessionStorage.sg_scope`、主題選擇、遊戲連結與題庫打包邏輯。
- 只調整 markup、CSS、ARIA 與 render 顯示狀態。

## 3. 所有遊戲統一的開始流程

適用：`tank`、`miner`、`bomb`、`runner`、`battle`、`ship`、`sniper`

### 3.1 固定資訊層級

所有遊戲開始畫面的選項區依下列順序排列：

1. 小型題庫狀態：`本次段考單字` 或 `系統內建單字`，若能取得單元名稱與字數一併顯示。
2. 遊戲特有的必要選擇（目前主要是 sniper 的「句子題／單字題」）。
3. `題目內容`（只有圖片題庫才顯示）。
4. `選擇難度`。
5. 單一、醒目的 `開始遊戲` 按鈕。

模式按鈕只負責選取，不可點下去立刻開始；所有遊戲都由最後的 `開始遊戲` 按鈕啟動。這能避免學生想改另一個設定時，遊戲已經先開始。

### 3.2 題目內容：取代「全圖檔模式」

不再顯示「全圖檔模式」這個名稱，也不使用單顆 checkbox。改成二選一 segmented control：

- `混合練習（推薦）`：對應 `sgAllPic = '0'`。
- `只玩圖片題`：對應 `sgAllPic = '1'`，下方小字說明「看英文、聽發音，選出正確圖片」。

規則：

- 只有題庫中有足夠圖片時才顯示整個「題目內容」區塊。
- 沒有圖片的段考題庫直接隱藏，不顯示 disabled 控制。
- 繼續共用既有 `localStorage.sgAllPic`，不得更換 key 或破壞跨遊戲記憶。
- 切換元件必須是真實 DOM `<button>` 或 radio，支援觸控、滑鼠與鍵盤。

### 3.3 難度選擇

標題統一為 `選擇難度`，顏色與圖示統一：

| 模式 | 樣式 | 一行說明 |
|---|---|---|
| 簡單 | 綠色、🌱 | 標準玩法，圖片題較多 |
| 一般 | 藍色、🙂 | 標準玩法，中英互相練習 |
| 困難 | 紅橘色、🔥 | 節奏更快，加入額外威脅 |

顯示規則沿用現況，但**只顯示可選的兩個主要模式**：

- 有圖片題庫：顯示「簡單／困難」。
- 無圖片題庫：顯示「一般／困難」。
- 不顯示灰色、無法點擊的第三顆按鈕。

模式卡片要有一致的 selected 樣式（外框、勾選圖示、`aria-pressed`），不可只靠顏色。手機至少 44px 點擊高度。

### 3.4 開始按鈕與 iOS 音訊

- 所有遊戲統一使用真實 DOM `<button>` 的 `click` 開始遊戲。
- `bomb` 目前是 canvas 選單，需保留真實 DOM overlay，但改成「選模式」與「開始遊戲」分開的按鈕。
- 第一次語音／BGM 解鎖仍放在 `開始遊戲` 的真實 click handler；不可改回 canvas touch 直接啟動。
- 開始按鈕未選難度時 disabled，並顯示短提示「請先選擇難度」。

## 4. 各遊戲特殊處理

### 4.1 tank.html

- 主要區只保留「題目內容」「選擇難度」「開始遊戲」。
- 目前獨立的「砲彈下墜：簡單／困難」不是學生進場的必要決策，移入預設收合的 `進階設定`。
- 不刪除 `tankDiff` 與砲彈重力功能，避免改變既有玩家設定。
- 開始提示改為按鈕操作，不再同時寫「空白鍵、滑鼠、輕觸都可開始」。遊戲開始後的鍵盤／觸控玩法不變。

### 4.2 miner.html

- 在現有 start card 內套用統一順序。
- 難度卡只做選取，保留現有 `▶ 開始遊戲` 作唯一啟動入口。

### 4.3 bomb.html / bomb-game.js

- canvas 仍可畫背景與標題，但設定控制與開始按鈕使用可存取的 DOM 元件。
- 畫面只呈現目前可選的兩個難度，不畫灰色第三個。
- 返回鍵移到右上後，重算左上 HUD／暫停鈕的位置；不要再以返回鍵右緣當左側 HUD 起點。

### 4.4 runner.html

- 保留現有獨立「開始遊戲」按鈕。
- `typePanel` 內改成統一的題目內容與難度區，不顯示不可用模式。

### 4.5 battle.html

- 難度卡說明過長，目前第一眼資訊量太大。卡片只留模式名稱與一行摘要。
- 詳細敵人秒數、金幣、BOSS 規則留在上方規則區，或隨選取模式更新 `difficulty-summary`，不要全部塞進三張卡。
- 保留「開始戰鬥」作唯一啟動入口。

### 4.6 ship.html

- 主要難度只顯示目前可用的「簡單／一般」其中之一，加上「困難」。
- `特殊模式` 不與主要難度並排，移到次要區塊 `其他玩法：特殊挑戰`，避免第一眼出現 3～4 個選項。
- 選擇特殊模式後仍由同一顆 `開始遊戲` 啟動。
- 暫停畫面的遊戲內 `回主選單` 改名為 `重新選擇模式`，避免和右上角「返回遊戲選單」混淆。

### 4.7 sniper.html

- 保留第一步「句子題／單字題」，但改成清楚的 segmented control。
- 接著才顯示題目內容與難度；不可用模式直接隱藏。
- 移除開始畫面底部的 `← 返回遊戲選單`。
- 唯一的中央站返回鍵固定在右上角，開始畫面與遊戲中都可見。
- 操控設定保留為次要按鈕，不與開始按鈕同層級。

## 5. 返回鍵統一

修改所有遊戲的中央站返回鍵：

- 固定右上角：`top/right` 加上 `env(safe-area-inset-top/right)`，避免 iPhone 瀏海與圓角。
- 文字統一：`← 遊戲選單`；`aria-label="返回遊戲選單"`。
- `z-index` 必須高於開始、暫停、結算 overlay，所有畫面都能使用。
- 觸控高度至少 44px，桌機可視覺較小但保留點擊範圍。
- 不遮住 HUD、暫停鍵、分數或題目；各遊戲需分別檢查 top bar 排列。
- sniper 刪除重複的底部返回鍵。
- 遊戲內部的「回本遊戲開始畫面」不得叫「返回遊戲選單」，統一改為「重新選擇模式」。

## 6. 修改檔案清單

除非實作時確認有必要，不新增共用 JS/CSS 檔；維持現有單檔遊戲架構。

## 7. 實作順序

1. 先做 `index.html` 的來源卡，確立配色與可存取狀態。
2. 做已有獨立開始按鈕的 `runner`、`miner`、`battle`，建立統一 UI 範本。
3. 套用到 `tank`、`ship`、`sniper`，處理各自的特殊選項。
4. 最後做 canvas 架構不同的 `bomb`。
5. 統一 7 個遊戲的右上返回鍵與文字。
6. 完成靜態自檢後 commit、依 `AGENTS.md` 先 fetch 比對遠端，再決定是否 push。

## 8. 交付前靜態自檢

依專案規範不自行開 server／Playwright；只做以下檢查：

1. `.js` 跑 `node --check`；每個改過的 HTML 抽出內嵌 script 後跑 `node --check`。
2. 確認所有 `apply*Data`、`normalize*Vocab`、hash、postMessage 注入鏈沒有改壞。
3. 用搜尋確認畫面上不再出現「全圖檔模式」。
4. 用搜尋確認每個遊戲的中央站返回鍵都是右上定位，sniper 沒有第二個返回鍵。
5. 確認 `sgAllPic`、各遊戲模式 localStorage key 沒有改名。
6. `git diff` 只包含本計畫的 UI 工作，不重排無關遊戲程式。
7. 檢查按鈕都是實際 `<button>`／radio，可用 click、觸控、Enter／Space 操作。
8. 檢查手機直式、橫式 CSS 不會讓開始按鈕或右上返回鍵被裁掉（只做程式碼靜態檢查）。

實作者交付後，請 Ivan 自測：

1. 從學生教材頁進主機：是否一眼看出目前是「本次段考單字」。
2. 切到內建題庫：是否一眼看出來源已換成藍色內建卡，主題區才出現。
