# 三模式改版計畫書（簡單／一般／困難＋全圖檔模式）

> 給實作者：**開工前先讀本專案根目錄的 `CLAUDE.md`**（測試規範、push 帳號、iOS 發音兩大地雷），並遵守全域 skill `studentgames-dev` 的架構慣例與交付前自檢。本計畫書是與 Ivan 討論定案的規格，實作時以此為準；規格有矛盾或做不到時，停下來問，不要自行腦補。

## 0. 背景與已定案的決策

Ivan 要把所有遊戲的模式統一成「簡單／一般／困難」三檔，依單字來源限制可選模式，並新增「全圖檔模式」開關。以下皆為與 Ivan 討論後的定案：

1. 模式可選性：**有圖題庫 → 只能選「簡單」「困難」；無圖題庫 → 只能選「一般」「困難」**，不可選的按鈕灰化（disabled）但仍顯示。
2. **玩法分級原則**：**簡單與一般的遊戲方式、參數鬆緊完全一樣**（同一套玩法、無額外威脅機制），兩者只差題型；**困難才改變遊戲方式**（加入反擊、被發現、砲塔等新機制或高壓參數）。各遊戲原本更鬆的簡單檔（tank 簡易幾乎不出怪、ship 無限彈、battle easy 檔）**廢除**，簡單一律採用該遊戲「一般」的玩法參數。
3. 全圖檔模式：獨立開關，放在**各遊戲開始前的選擇畫面**，只在有圖題庫（內建單字）時顯示，對**簡單與困難都有效**：
   - **開啟**：不論簡單或困難，每題都是「英文＋發音 → 選圖片」。
   - **關閉**：混合題型——簡單＝圖英題為主、穿插約 1/3 中英題；困難＝中英題為主、穿插約 1/3 圖英題。
   - 一般模式不受開關影響（固定中英互考，開關對它不顯示作用）。
4. **開始畫面流程**（三步）：① 遊戲特有的前置選擇（目前只有 sniper：選「句子題」或「單字題」）→ ② 全圖檔開關（有圖題庫才顯示）→ ③ 模式選擇（簡單／一般／困難，依來源灰化）。
5. **圖片題方向鐵則**（Ivan 明確指定）：一律「上面顯示英文單字＋唸發音，下面的選項／要打的目標是圖片」，**絕不反過來**（不出「看圖選英文」）。
6. **絕不出「圖↔中」題**（看圖選中文、看中文選圖都禁止，考了沒意義）。
7. 射擊類遊戲（tank／ship／sniper）的「英→中」題：**要做**——目標（樹上字牌、船身、大樓招牌）可以掛中文，也可以掛英文。
8. ship 新增「一般模式」＝簡單同一套玩法：**彈藥有限（每回合 4 發）、無反擊、16 秒、答錯（打錯船）時目標船加速**；**特殊模式原封不動保留**成第四顆按鈕。
9. sniper：**第一步先選「句子題」或「單字題」**；新增單字版「簡單模式」＝**音圖題**（題目卡顯示英文＋唸發音，射掛著對應圖片的目標）。
10. 主機 `index.html` 不動（模式與開關都在各遊戲內）。

## 1. 共用設計（所有遊戲一致）

### 1.1 題庫來源判斷
- 「內建單字」＝題庫帶 emoji（wordbank 主題經主機 `lessonData` 注入，或遊戲內建 VOCAB 有 emoji）。
- 「連結單字」＝`lessonData` 的字沒有 emoji（學校段考單字）。
- 判斷方式沿用 runner 既有做法：**pool 中帶 emoji 的字 ≥ 4 → 視為有圖題庫**。
- 注意：主機選內建主題也是走 lessonData hash 注入，所以「內建 vs 連結」實際上只能用「有沒有 emoji」判斷，這是正確且唯一可靠的依據。

### 1.2 各模式題型與玩法
| 模式 | 可用來源 | 題型（全圖檔關） | 題型（全圖檔開） | 玩法 |
|---|---|---|---|---|
| 簡單 | 有圖題庫 | 「英文＋發音→圖」為主，穿插約 1/3 中英題 | 全部「英文＋發音→圖」 | **與一般完全相同** |
| 一般 | 無圖題庫 | 中→英、英→中（開關不適用） | — | 標準玩法（無額外威脅機制） |
| 困難 | 兩種來源 | 中→英、英→中為主；有圖題庫再穿插約 1/3「英文＋發音→圖」 | 全部「英文＋發音→圖」 | 各遊戲現行困難檔（新威脅機制／高壓參數） |

- **簡單與一般共用同一套玩法參數**（＝各遊戲現行「一般」檔）；各遊戲原本更鬆的簡單檔一律廢除。
- ship／sniper 既有的句子挖空題保留，歸屬在一般／困難的題型輪替中（有例句才出，現行邏輯；sniper 另有題型前置選擇，見 §2.7）。
- 圖片題（含穿插的）永遠遵守 §0-5 方向鐵則、§0-6 禁圖中。

### 1.3 全圖檔開關
- UI：開始畫面上的 checkbox／切換鈕，**只在有圖題庫時顯示**，位置在模式按鈕之前（§0-4 流程），文案例：「🖼️ 全圖檔模式（每題都是英文＋發音 → 選圖片）」。
- 對簡單與困難都生效（開＝純圖英；關＝混合，比例見 §1.2）；一般模式不受影響。
- 儲存：`localStorage` key **`sgAllPic`**（`'1'`/`'0'`），**所有遊戲共用同一把鑰匙**（同網域），Ivan 在某個小朋友的裝置上開一次，該裝置所有遊戲全部生效。
- 各遊戲自己的模式選擇也用 localStorage 記住（各自的 key，如 `tankMode`、`runnerMode3`…），載入時 clamp 到該題庫來源允許的模式。

### 1.4 發音（重要，別踩雷）
- 現況：**tank（簡易模式）、miner、bomb、runner、sniper 已有 speechSynthesis；ship、battle 完全沒有**。
- ship、battle 要新增發音時，**必須套用專案 CLAUDE.md「iOS 兩大地雷」修法**：
  1. 第一次 speak 由真實 DOM `<button>` click 解鎖（兩款的開始／模式按鈕都是真實 DOM 按鈕，可直接在點擊 handler 內解鎖）；
  2. 唸字時 `AudioContext.suspend()` 讓出 iOS 音訊工作階段、`onend/onerror` 時 `resume()`，加世代 token 防 race＋setTimeout 安全網。現成範本：`tank.html` 的 `speakWord`（約 965 行）、`bomb-game.js` 的做法（commit cb8764a）。
- 圖片題**一律同時顯示英文字＋發音**：發音在某平台失效時，小朋友看字仍可作答。
- 其餘 speechSynthesis 通則見專案 CLAUDE.md「跨平台相容標準」。

### 1.5 UI 慣例
- 三顆模式鈕永遠都畫出來，不可選的加 `disabled`＋灰化（`opacity:.32; filter:grayscale(1); cursor:not-allowed`），旁邊給一行提示文字，例：「內建圖片單字：可選簡單或困難」／「連結單字（無圖片）：可選一般或困難」。
- 樣式貼合各遊戲既有按鈕風格，不重排既有版面。

### 2.7 sniper.html（海岸狙擊手之狙擊鏡）
- 現況：簡單／困難都是**句子挖空**（無例句時 hint 模式＝顯示中文意思），差別在玩法（困難有「開槍沒中被發現」機制）；大樓樓層目標掛霓虹英文字；`speakWord` 已有。
- **開始畫面改成三步流程（Ivan 指定，sniper 是目前唯一有第一步的遊戲）**：
  1. **選「📝 句子題」或「🔤 單字題」**。句子題需要題庫有例句（內建 wordbank 有；學校連結單字通常沒有）——例句不足時「句子題」鈕灰化、自動選單字題。
  2. 全圖檔開關：**選「單字題」且有圖題庫**時才顯示（句子題與開關無關）。
  3. 模式選擇（灰化規則見下）。
- 模式灰化＝題型 × 來源：
  - 句子題：**簡單永遠灰化**（句子沒有簡單版）；一般／困難可選（＝現行兩檔的句子挖空玩法，一般無被發現、困難有）。
  - 單字題：有圖→簡單＋困難；無圖→一般＋困難。
- 玩法參數：**簡單＝一般＝現行 easy 檔**（時限、機會數相同），差別只在題型；困難＝現行 hard 檔（開槍沒中被發現機制）。
- 單字題題型：
  - 簡單＋開：情報卡顯示英文＋🔊 唸發音（給重聽鈕），**樓層目標掛 emoji**，射對應圖片。
  - 簡單＋關：英→圖回合為主＋約 1/3 中英回合。
  - 一般：hint 雙向輪替——中→英（現行：顯示中文意思射英文）＋英→中（情報卡顯示英文＋發音，**樓層掛中文**）。
  - 困難＋關：中英雙向為主，有圖再穿插約 1/3 英→圖回合（被發現機制照常）。
  - 困難／簡單＋開：全部英→圖回合。
- 樓層霓虹字改畫 emoji／中文時注意寬度與字級。

## 4. 風險與注意事項

- **canvas 畫中文**（tank 樹牌、ship 船身、sniper 招牌）：字型 fallback、字級、寬度要實測，中文字比英文寬。
- **ship／battle 新增發音**是本次最大雷區，嚴格照 §1.4 與專案 CLAUDE.md 執行；做完標註「待 iOS 實機驗證」。
- 穿插比例「約 1/3」是體感值，不必精確，但不能變成一半以上。
- 各遊戲直接單開（無 lessonData）走內建 VOCAB：有 emoji 的視為有圖題庫，沒有的視為無圖，邏輯同 §1.1，不要另開特例。
- 改動一律貼合各檔既有風格；**不要引入共用 JS 檔**（維持每遊戲單檔自足的架構）。
